## 本次修改摘要

1. 将 `modules/planning/scripts` 的真实实现按功能分类：
   - `scripts/core/plan_pipeline.py`：核心 plan 生成、修复、渲染、执行合同。
   - `scripts/tools/planning_tools.py`：Planning 私有工具集合。
   - `scripts/blockers/build_blocker_action_plan.py`：阻塞行动计划。
   - `scripts/actions/propose_next_actions.py`：下一步行动建议。
2. 根目录旧脚本路径保留为兼容入口：
   - `scripts/plan_pipeline.py`
   - `scripts/planning_tools.py`
   - `scripts/build_blocker_action_plan.py`
   - `scripts/propose_next_actions.py`
   这些入口通过执行分类目录中的真实实现保持旧 import/直接执行路径可用。
3. 增强 `scripts/core/plan_pipeline.py`：
   - fallback plan 不再只是泛化步骤，新增环境证据、repo/data 需求、基线/消融、指标、失败分析、go/no-go、Claude Code handoff、论文证据检查等字段。
   - LLM/Claude 风格提示词要求输出完整可执行研究合同，避免提前绑定具体 repo_path、数据路径、训练命令或 Environment 尚未确认的 base。
   - 每次写 plan 时同时生成 `experiment_plan.json` 和 `taste_plan_bridge.json`。
   - `experiment_plan.json` 在没有唯一 `selected_plan_id` 时保持 blocked，后续模块不得执行。
4. 增强 `modules/planning/main.py`：
   - 保持原 `--run-id` 框架调用兼容。
   - 新增 standalone 输入：`--idea-json`、`--idea-md`、`--idea-text`。
   - 支持 `--backend claude_code`，由 planning 内部 Claude runner 调用 nvm 环境中的 Claude CLI 生成/评估/修复 plan。
   - standalone 默认输出到 `modules/planning/runs/<run-id>`，不会同步到 planning 外的 runtime/latest/project state。
   - `main.py` 的模块入口不再主动把 `web/backend` 加进 PYTHONPATH；Planning 入口保持后端自足。
5. 新增 `modules/planning/.gitignore`，排除 `runs/`、`.tmp/` 等运行中间产物。
6. 重写 README，补充中文的人类可读使用说明、输入输出口径、流程逻辑、脚本分类和维护规则。
7. 重建 `script_manifest.json`，保留 4 个兼容入口作为脚本清单项，并记录各自真实实现路径。

## 重要边界

- 严禁为了让 plan “看起来可执行”而写死具体 `repo_path`、`local_path`、`selected_repo_path`、`training_script` 或命令；这些属于 Environment/Experiment 的权威范围。
- Planning 可以写“候选 artifact/data/protocol 需求”，不能写“已选中某 repo/数据/命令”。
- `selected_plan_id` 为空或多选时，`experiment_plan.json` 必须保持 blocked。
- standalone 调试输出只能放在 `modules/planning/runs/` 或 `modules/planning/.tmp/`，不要写到 `/tmp`、仓库根、runtime、web 或其它模块。
- 修改脚本结构时要维护 README 和 `script_manifest.json`，并保持根兼容入口存在。
