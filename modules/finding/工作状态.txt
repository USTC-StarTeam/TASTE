【模块一句话定位】
  TASTE 七阶段中的 Find 阶段后端：从"研究主题 + 研究兴趣 + 研究者画像 + 来源选择"
  出发，抓取候选文献/代码线索 → 多级筛选打分 → 产出最符合研究者要求的推荐论文/资料。
  契约：只做 Collect / Filter / Score / Rank；不做全文精读（全文证据属于 reading 模块）。

  公开入口只有一个：modules/finding/main.py
    - `python modules/finding/main.py --contract`          打印模块契约（零依赖，可当冒烟测试）
    - `python modules/finding/main.py --action find ...`   跑完整 Find 流水线
  scripts/ 下都是模块私有后端，不应被前端直接拼路径调用。

 二、架构现状（2026-06-18 全量梳理结论，重要！）
--------------------------------------------------------------------------------
  finding/scripts 里其实有"两条互不调用的发现路径"，分清楚是维护第一要务：

  路径 A —— 真正的 Find 流水线（主线，对应契约与网页 Find）
    main.py --action find → find_pipeline.run_find(FindRequest(config, selection))
    用 find_support.py 里的真实 fetch_* 适配器抓取，按 VenueSelection.include_* 开关取数。
    覆盖：arXiv / bioRxiv / GitHub / HuggingFace / Nature / Science / 会议-按年(DBLP/OpenReview/
          NeurIPS/ICML/PMLR/CVF/ECCV/ACL)。Semantic Scholar / OpenAlex 仅用于补摘要(enrich)。
    独立入口默认产物写到 modules/finding/.runtime/runs/<run_id>/（find_results.json / article.md / source_status.md / 各类报告）；外部框架若显式设置 WORKFLOW_RUNTIME_DIR，则尊重外部运行根。

  路径 B —— 项目耦合的 discover 脚本（旧/并行线，给项目工作流用，不是独立 Find）
    discover_arxiv.py / discover_semantic_scholar.py / discover_github_repos.py / ingest_discovery.py
    需要 --project，import project_paths，把结果写进 projects/<id>/discover|state|reports|raw/papers|wiki_*。
    只被 framework/scripts/run_project.py 与 modules/environment 调用，find_pipeline 不调用它们。
    维护提醒：只看 discover_*.py 会误以为 bioRxiv/HuggingFace 没实现——其实在路径 A 的 find_support 里。

  缓存构建器：build_venue_metadata_cache / build_openreview_cache / build_category_summary /
              update_local_database → 写 runtime/local_database/<venue>/<year>/。
  质量/工具：finding_quality_tools（plan_literature/paper_quality/literature_base_candidates）、
              run_literature_base_audit、build_literature_tool_packet、run_literature_tool —— 多为项目耦合。
  data/：ccf_venues.json(649 会议/期刊) + default/custom_venues.json + quality/*.json(334 期刊)。
         这些是被跟踪的"输入数据"，不是产物。quality/*.json 只影响"打分加成"，不决定"抓哪些"。

  依赖面 / 独立性结论：

 三、关键约束：框架的"模块边界契约"（动结构前必须懂，由测试强制）
--------------------------------------------------------------------------------
  tests/test_module_boundaries.py 强制以下规则，任何结构改动都要让它（及全部 750 测试）保持绿：
  1. finding/scripts 目前是"扁平"的：脚本归属按【文件名】在
     framework/scripts/auto_research/module_boundaries.py 的 SCRIPT_OWNER_OVERRIDES + classify_script 里登记；
     每个模块的 script_manifest.json 必须与扁平 .py/.sh 文件集合"精确相等"。
  2. tests/path_helpers.py 的 script_dirs() / _migrated_scripts() 用【非递归 iterdir】只看 scripts/ 顶层文件；
     script_path()/load_script() 也只在顶层找 → 把脚本移进子目录会让它们"消失"，并破坏扁平 import 与 manifest。
  3. test_find_to_plan_stage_modules_stay_flattened 禁止 finding 下出现名为
     auto_research/auto_find/... 的子目录（但 sources/ 这类普通子目录不在禁止名单里）。
  4. test_seven_stage..._single_main_entrypoint：模块根必须有 README.md/__init__.py/main.py/script_manifest.json；
     不能有 cli.py/contracts.py；main.py 要有 STAGE_NAME='finding' 与 def contract()；__init__.py 要
     `from .main import contract, main`。
  5. test_finding_reading_full_text_responsibility_boundary：find_pipeline.py 不得含全文可读性门控词
     （full_text_readability_gate 等）—— 全文是 reading 的事。
  → 含义：把脚本"真分子文件夹"必须【同时】改框架边界机器（path_helpers/module_boundaries/manifest生成/main.py路由）
     与对应测试，且要让 7 个 stage 模块的边界测试都还绿。

  巨型文件的"拆分公共面"约束（动它们前必须懂）：
  * find_pipeline.py（7938 行）唯一对外符号是 run_find；被 main.py 和 framework/scripts/run_frontend.py import。
    其余全是下划线私有。→ 只要 `from find_pipeline import run_find` 仍可解析，内部随便重排都对外透明。
  * find_support.py（6502 行）其实是 8 个旧模块的"机械拼接"，文件内还留着 banner 注释；
    原始干净版就在 third_party/reference_TASTE_latest/auto_research/auto_find/（catalog/local_index/local_cache/
    local_rank/quality/category_select/profile_normalize/sources）。→ 拆分=按 banner"反拼接"，不是凭空切。
    对外有 54 个被消费符号（find_pipeline、web server、build_*、tests），其中 5 个是下划线私有(被 tests import)，
    且 tests 会做 `import find_support as sources; sources.requests=...; sources.fetch_*=...` 的 monkeypatch
    → 任何拆分都要保证这 54 个符号 + `requests` 仍能从 find_support 这个名字 import 到（用 re-export 门面解决）。
